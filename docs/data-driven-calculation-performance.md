# Data-driven calculation performance

Date: 2026-07-16

This report continues from `docs/mutation-archetype-performance.md`. It treats the earlier choice identity, batching, targeted catalog/list/card updates, conflict indexing, structural reconciliation batching, selective invalidation, derived-request coalescing, and persistence coalescing as established. Those paths were used as controls, not re-investigated.

Desktop measurements use Chromium without CPU throttling. Mobile Chromium uses Pixel 7 emulation with 4x CPU throttling. A p95 is shown only for five-sample runs. All other measurements are explicitly marked diagnostic and report the observed sample, not a percentile.

The earlier report remains the baseline for choice/variant identity, choice cancellation, identical-variant batching, multi-buy, simple quantity batching, targeted catalog/character cards, conflict replacement, indexed conflict sources, grant/cascade batching, selective invalidation, worker coalescing, and the one-commit/one-refresh/one-persistence shape. Lightweight controls confirmed those mechanisms remain active:

- conflict replacement used no full render and spent 1.9 ms in grant planning, 1.9 ms in requirement evaluation, 1.6 ms in five conflict evaluations, and 0.1 ms in replacement planning;
- a structural artifact cascade used a targeted render and one commit; its remaining rule work is discussed below;
- the full unit, rule-validation, data-validation, mutation, parity, mobile, and PWA suites remained regression protection rather than discovery workloads.

The investigation did not repeat the already documented catalog-add and structural-list profiling unless a control was needed to attribute a newly instrumented shared stage.

## 2. Behavior-signature inventory

The audit normalizes each catalog entry into four parts:

1. structure: catalog types, stackability, individual/top-level shape, container, vehicle, bundle, catalog/custom identity, and runtime UID requirement;
2. variant behavior: choice fields and duplicate policy, variant-defining metadata, quality support, artifact binding, snapshot, and hidden/revealed state;
3. rules: requirements, grants, conflicts, modify targets, list-wide dependencies, and inventory/money grants;
4. expected impact: inventory, list, economy, traits, combat, effects, capacity, corruption, artifacts, snapshots, visibility, persistence, or unknown.

An action signature combines the entry signature with the production mutation (`add`, quantity change, removal, level change, quality change, movement, load/unload, reveal/snapshot, reload) and its expected impact. The current catalog produces 148 distinct entry signatures and 816 distinct action signatures across 1,538 entries. Every group, member count, deterministic representative set, and expected impact is retained in the machine artifact.

Representative selection is deterministic: first by frequency, then greatest relevant rule/state count, then a rare classified feature, then a measured outlier when one exists. Names are test labels only; production classification does not branch on them.

| Overlapping behavior family | Catalog entries | Representative behavior | Expected impact | Current path |
|---|---:|---|---|---|
| Simple local | 1,047 | Stable catalog row with no structural rule coupling | Row/list entry, totals when inventory, persistence | Targeted existing path. |
| Choice | 16 | Stackable and individual choice variants | Variant identity, row/structure, declared rule domains | Targeted when identity and impact are stable. |
| Variant-bearing | 155 | Choice, quality, artifact, or other variant metadata | Row/structure, economy, declared derived domains | Quantity and supported quality changes are targeted; ambiguous metadata falls back. |
| Batch/multi | 14 | Multi-buy, bundle/grant batch, repeated variant | Union of affected rows/domains | One root batch/commit; unknown mixed topology falls back. |
| Rule/reconciliation | 419 | Requirement, grant, conflict, or modify rules | List structure, grants, declared derived domains | Existing indexed/batched path; list-wide and ambiguous rules remain conservative. |
| Coupled list/inventory | 152 | Inventory grants, artifact/list coupling, snapshots | List, inventory, economy, artifacts/snapshots | Known grants are batched; artifact/snapshot ambiguity falls back. |
| Complex inventory topology | 276 | Individual items, containers, vehicles, bundles | Inventory structure/totals and persistence | Stable top-level row changes may be targeted; topology mutations use the full path. |
| Custom/legacy/unknown | 0 static entries; 76 import mappings | Runtime custom/imported rows | Unknown | Full safe path unless identity, schema, and impact are all proven. |

Special signature counts are: 8 stackable choices, 2 snapshot entries, 321 hidden/revealed entries, 2 bundles, 10 vehicles, 13 containers, and 261 individual entries. Catalog data contains no entry without an ID; missing IDs are a runtime/import concern.

Known modify targets are derived from the runtime `registerMal` dispatch table, not catalog names or a second hand-maintained list. Rule results are cached only on the current entry object and canonical level. Cache records include the catalog array reference, size, and `__entryDataVersions.db` generation; a changed reference or generation creates a fresh bounded per-entry map. Character-dependent requirement/conflict decisions are not retained across mutations.

## 3. Coverage gaps

The new real-UI scenarios close the largest gap for existing metadata-bearing rows: quantity increase/decrease, final-copy removal, and an economy-only quality addition now use their production buttons and popups. They cover warm and unavailable aggregate snapshots, 250 unrelated inventory rows, active filters, forced safe/optimized comparison, persistence flush, and reload.

Remaining gaps are:

- artifact reveal/hide/bind and snapshot synchronization: no reliable scenario yet follows the complete production UI action through all coupled state;
- bundle removal and nested movement: no real-UI diagnostic completed; retained simulations do not prove production-handler cost;
- container deletion/unwrap and specialized vehicle removal: retained scenarios mutate through test code; only real vehicle load/unload is performance-mapped;
- quality removal and declared combat/trait/effect quality targets: correctness paths exist, but only economy-only add/remove has forced-safe parity and reliable performance attribution;
- large specialized character switching: correctness exists for ordinary switching, but the new large-state scenario did not produce a reliable completed sample;
- structural level changes: existing mapped optimization remains a control, but the attempted new mobile diagnostic did not complete reliably enough to publish;
- Mobile WebKit performance: correctness parity passes, but no reliable timing was collected for the changed scenarios; Mobile Chromium was the diagnostic device because CPU/render amplification dominated;
- PWA: correctness and interaction budgets pass. No evidence identified a service-worker-specific mutation bottleneck, so no separate PWA performance matrix was run.

The audit marks old custom-entry, money, heavy-save, generic inventory removal, container, vehicle-removal, tag-removal, and inventory-clear simulations as `scenario-does-not-match-real-user-action`. They remain useful correctness/setup helpers, but not evidence about click latency.

Instrumentation now attributes rule source lookup/normalization, requirements, conflicts, grants/dependents, list-wide checks, inventory normalization/path/UID/flattening/aggregates/artifacts, derived request/queue/worker/apply, rendering/sort/card/DOM work, persistence preparation/materialization/transaction, hidden-surface refreshes, and canonical fallback reasons. No second profiler was introduced.

## 6. Data-driven root causes

The root cause was not one catalog entry. It was the combination of a simple UI mutation with metadata or topology that caused the old save/render path to assume broad impact:

- choice metadata plus quantity entered inventory normalization, full UID traversal, artifact scanning, full sort/group/card rebuild, and an unrelated derived request even when the existing variant identity did not change;
- final-copy removal correctly meant `inventory.structure`, but rendering treated every structural invalidation as requiring a full inventory rebuild. Semantic impact and render strategy had been conflated;
- quality metadata used the same broad path even when every runtime-registered modify target declared only price/weight/economy impact;
- active filters make membership/order postconditions ambiguous, so a known row cannot yet be patched safely;
- nested inventory, containers, vehicles, and bundles change parentage, capacity/category membership, or multiple rows. Their local postcondition is not currently provable;
- artifact plus list/snapshot/hidden state can couple an inventory row to list entries, revealed IDs, effects, and snapshot sources;
- grant plus requirement/dependent removal is correctly broader than a row mutation; dependent discovery is measurable but remains behaviorally significant;
- unknown modify targets, manual overrides, custom entries, and legacy metadata cannot safely declare a narrow domain;
- a missing aggregate snapshot forces a complete totals rebuild even though rendering remains local.

Mobile makes the fallback disproportionately worse because JavaScript sorting/model construction, DOM creation/replacement, and layout all amplify under 4x CPU throttling. Persistence transactions also grow on mobile, but they did not block first visible feedback and were much smaller than rendering.

## 7. Implemented general improvements

### Mutation-impact contract and semantic/render separation

`js/inventory-utils.js`, `js/store.js`, and `js/main.js` now carry a single internal impact contract with `local`, `declared`, or `fallback` classification; affected semantic domains; identity and topology guarantees; aggregate strategy; render strategy; derived invalidation; and exact fallback reasons.

Semantic invalidation is not weakened to obtain a fast render. A known top-level final removal still invalidates `inventory.structure`, `inventory.totals`, `summary.economy`, and persistence, while independently selecting `targeted-remove`. Different render strategies within one batch escalate safely to full rendering.

### Stable top-level quantity and final removal

`js/inventory-utils.js` updates an existing row without recreating variant identity, validates only the affected UID (or the expected absence after removal), applies price/weight/food/quantity deltas, patches the known card and totals, and retains one common commit/refresh/persistence schedule. It skips full quality normalization, vehicle partitioning, artifact scans, flattening, and derived work only when classification proves they cannot change.

Cold aggregates use the exact `aggregate-snapshot-unavailable` reason and rebuild totals without turning the mutation into a full render. Active filters, unstable identity, missing UID, nested/container/vehicle/bundle topology, artifact/snapshot coupling, rules, custom/legacy data, and DOM postcondition failures retain precise full fallbacks.

The reliable measurements in section 4 show the result. On Mobile Chromium, quantity visible p95 fell from the forced-safe 933.9 ms to 117.3 ms and complete-consistency p95 from 1,667.0 to 167.7 ms. Final removal visible p95 fell from 1,771.2 to 421.7 ms and consistency p95 from 2,553.0 to 493.4 ms.

### Declared economy-only quality changes

The second optimized family is quality/metadata mutation whose runtime-registered targets declare only inventory row/totals/economy impact. `js/inventory-utils.js` now patches quality chips, details/info, price, weight, and totals; applies a before/after aggregate row delta; and avoids the full render/artifact/derived pipeline.

Known combat, trait, effect, or summary targets currently retain `declared-quality-impact-unoptimized`. Artifact, snapshot, structure, custom, manual, legacy, unknown target, nested path, active filter, and missing UID cases retain the safe path. No representative name appears in production conditions.

On Mobile Chromium, economy-quality visible p95 fell from the forced-safe 650.7 ms to 81.4 ms and consistency p95 from 992.8 to 108.2 ms. Desktop visible p95 fell from 143.6 to 105.4 ms; the larger p95 variance in the optimized sample is presentation scheduling, while its median fell from 140.4 to 37.0 ms and its attributed surface work from 127.0 to 1.9 ms.

### Rule cache lifecycle and attribution

`js/rules-helper.js` now records source resolution, cache lookup/hit/miss, normalization, requirements, conflicts, grants, dependents, cascade checks, modify-target mapping, and list-wide checks. The existing entry/level cache is guarded by catalog reference, size, and generation. Measurements showed high warm hit rates and sub-2 ms normalization in the conflict control, so no speculative cross-mutation requirement/conflict cache was implemented.

### Rendering, persistence, and profiling

`js/perf.js`, `js/persistence.js`, `js/index-view.js`, and `js/inventory-utils.js` extend `window.symbaroumPerf` with user-visible milestones, outer-stage timing, inner call counts, DOM/card work, hidden surfaces, persistence preparation/transaction/flush, and canonical fallback counts. `scripts/run-scenario-metrics.mjs` supplies the real-UI scenarios, device/state dimensions, cold aggregates, filters, and forced safe controls. `scripts/audit-mutation-archetypes.mjs` supplies the versioned ledger/signatures and retains the measurements.

### Correctness parity and cache invalidation

`tests/inventory-mutation-parity.spec.js` executes the real production controls twice from the same normalized state: first with the forced safe path, then with the optimized path. It compares inventory topology and row identity, variant metadata and quantity/qualities, money, artifact effects, derived state where applicable, rendered cards/totals, persistence flush, and reload. Quantity/final-removal and economy-quality add/remove pass in Desktop Chromium, Mobile Chromium, and Mobile WebKit. Character switching is not part of these cases because the optimized paths introduce no character-dependent cache and issue no derived worker request.

Generated legacy bundles were updated only after source changes and targeted tests.

Forced-safe controls account for 30 recorded fallback activations but are excluded from production-frequency ranking. Among observed production reasons, the retained artifact ranks `aggregate-snapshot-unavailable` (3), `vehicle-topology` (3), and `active-filter-membership-uncertain` (2). The counts describe this diagnostic corpus, not population usage frequency.

Recommended future order is:

1. same-device vehicle/container/nested scaling and topology parity, because it has the largest measured mobile latency;
2. filter-membership classification for quality/row changes, because the fallback is common and its rendered set is already bounded;
3. cold aggregate profiling in real sessions before considering a narrower rebuild/index;
4. scaled grant/requirement dependent discovery before adding another rule index or mutation-scoped decision cache;
5. artifact/snapshot real-UI mapping, then large specialized character switching;
6. custom/legacy optimization only after stronger schema and identity guarantees.

Insufficiently understood areas remain explicitly marked `unmapped`, `device-gap`, or `scenario-does-not-match-real-user-action` in the ledger rather than being presented as optimized.

## 9. Plain-language conclusion

The genuinely slow behavior was a mismatch between what the user changed and what the application rebuilt. Changing one quantity, removing one known row, or adding an economy-only quality could rebuild and replace an entire 250-row inventory, rescan artifacts, normalize every UID, and ask the derived worker to run. Mobile magnified that broad JavaScript and DOM work into roughly one to two seconds of visible or consistency delay.

The application now decides semantic impact separately from how the UI should render it. A final removal is still truthfully structural, but a known top-level card can be removed directly. Stable metadata quantities keep their identity and update aggregates by delta. Economy-only quality changes patch only the affected card and totals. These changes cover broad data-driven signature classes and contain no entry-name special cases.

The second discovery pass found that vehicle load/unload is now the largest measured remaining user-facing cost. It remains conservative because parentage, capacity, category, and nested layout are not yet provably local. Artifact/snapshot coupling, bundles, containers, nested inventory, unknown rules, custom/legacy rows, and active filters likewise retain explicit safe fallbacks.

Users should feel the largest improvement when adjusting metadata-bearing quantities, removing their final copy, or adding/removing economy-only qualities—especially in large mobile inventories. Structural vehicles and other ambiguous specialized states are documented in the order they should be investigated next, without trading correctness for a benchmark.
